# Plan Data Model

`plan.yaml` is the single source of truth for a tech-mentor planning session.
Markdown views (`dialogue`, `annotated`, `unified`) are deterministic
projections rendered by `scripts/render_plan.py` — never hand-written, never
regenerated by inference once the data exists.

## Why a data model instead of a document

Three formats used to mean three separately-maintained `.md` files that could
drift from each other, each requiring the LLM to re-infer structure from
prose to switch views. With one YAML file as the data and the formats as
pure functions over it, switching views is instant, deterministic, and never
fabricates content.

## Schema

```yaml
title: string
version: float                  # bump on every refinement
last_updated: "YYYY-MM-DD"

mentors:
  - id: string                  # short handle, e.g. "karpathy"
    name: string                # display name
    focus: string                # one-line perspective summary

components:
  - id: string                  # short handle, e.g. "calibration_module"
    title: string                # display title

contributions:
  - id: string                  # unique, e.g. "c1", "c2" — referenced by other contributions
    component: string            # must match a components[].id
    mentor: string                # must match a mentors[].id
    version: float                # when this contribution was added
    type: proposal | extension | challenge | response
    in_reply_to: string | null    # id of the contribution this responds to, if any
    depends_on: [string]          # ids this contribution's reasoning depends on
    summary: string               # one line, used in dialogue-view attribution tags
    body: |                       # full prose — the mentor's actual reasoning, untouched
      ...
```

## What goes in metadata vs. body

**Metadata fields** (`type`, `depends_on`, `in_reply_to`, `component`) are
cheap, structural facts the mentor persona declares while writing — not
content to invent. They answer "where does this fit" and "what does this
depend on," not "why is this true."

**`body` stays free prose.** Do not decompose it into `what`/`why`/`tradeoffs`
sub-fields. That looks like more structure but actually invites two failure
modes: empty fields filled just to pass validation (structure as theater),
and reasoning that gets chopped into boxes it doesn't naturally fit —
trade-offs are usually argued inline with the proposal, not bolted on after.
If you need to flag that a contribution contains a trade-off worth finding,
that's what `type: challenge` or a short `summary` is for — a pointer, not a
duplicate of the content.

## `type` vocabulary

- `proposal` — first contribution to a component, starts the thread
- `extension` — builds on a prior contribution without disputing it
- `challenge` — surfaces a problem, risk, or unresolved trade-off in a prior
  contribution (these are what `query.sh ... type challenge` finds — the
  wizard's "fault lines," informally, inside tech-mentor)
- `response` — addresses a specific `challenge`

## Operations

| Operation | Tool | Why it matters |
|---|---|---|
| `validate` | `scripts/validate.sh plan.yaml` | catches broken `depends_on`/`in_reply_to`/`component` refs before render |
| `query` | `scripts/query.sh plan.yaml <field> <value>` | e.g. find all open challenges: `type challenge` |
| `render` | `scripts/render_plan.py plan.yaml --format X` | deterministic view switch, no LLM call needed |
| `diff` | compare `contributions[].id` sets between two versions | what's new since last refinement, not a text diff |

Run `validate` after every refinement, before render. It's cheap and catches
the most common authoring mistake: referencing a contribution id that was
renamed or never created.
